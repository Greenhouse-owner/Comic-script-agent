## 输入

- **用户原始输入**：模糊或不完整的文本
- **当前上下文**（可选）：已有的项目信息、对话历史摘要
- **需要澄清的维度**（可选）：如"风格"、"角色数量"、"章节数"

## 输出格式

```markdown
## 需要确认

{简要说明为什么需要澄清}

1. **选项A**：{描述}
2. **选项B**：{描述}
3. **选项C**：{描述}
4. **自定义**：请输入你的想法

> 请输入选项编号（1-4）或直接描述你的想法
```

## 硬约束

1. **选项数量 3-4 个**（含"自定义"选项）
   - 太少没有选择空间，太多造成选择困难
2. **选项必须互斥**
   - 不能有重叠的选项
3. **必须包含"自定义"选项**
   - 用户永远有权跳出预设选项
4. **选项描述要具体**
   - 不能是"好的"/"不好的"这种空泛选项
   - 每个选项要让用户能想象出具体效果
5. **语言风格友好**
   - 不要让用户感觉被考试，而是在帮助他们理清想法

## 失败处理

- **上下文不足以生成有意义的选项**：返回开放式问题而非选择题
- **用户已经给出足够信息**：返回 `[无需澄清]`，建议直接进入下一步
- **维度过多**：每次只问一个维度，标记其余维度为 `[待后续澄清]`
